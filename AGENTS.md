# AGENTS.md

This file is guidance for agentic coding assistants working in this repo.
It summarizes build/test commands and code style conventions observed in the codebase.

## Scope

- Applies to the entire repository unless overridden by a deeper `AGENTS.md`.
- No Cursor rules were found in `.cursor/rules/` or `.cursorrules`.
- No Copilot rules were found in `.github/copilot-instructions.md`.

## Quick Context

- Strato has three main Swift packages:
  - `control-plane` (Vapor web app)
  - `agent` (CLI hypervisor agent)
  - `shared` (shared models + WebSocket DTOs)
- Reusable libraries live in `libs/` as separate SPM packages:
  - `libs/swift-ovn` — OVN/OVS networking (used by agent, Linux only)
  - `libs/swift-qemu` — QEMU hypervisor bindings (used by agent)
  - `libs/swift-scim` — SCIM protocol implementation (used by control-plane)
  - `libs/swift-ssf` — Shared Signals Framework receiver (not yet wired up)
- Development flows are often orchestrated via `Taskfile.yml`.

## Build Commands

### Task Runner (recommended)

- `task dev` — start the full local dev environment (DB, SpiceDB, control-plane, agent).
- `task start-control-plane` — build and run control-plane.
- `task start-agent` — build and run agent.
- `task status` — show running service status.
- `task stop` — stop all services.
- `task clean` — remove containers and cleanup.

### Control Plane (Swift/Vapor)

- `cd control-plane && swift build`
- `cd control-plane && swift run`
- `cd control-plane && swift test`
- `cd control-plane && swift test --filter <TestCase>/<testName>`
- `cd control-plane && vapor serve` (if Vapor CLI installed)

### Agent (Swift)

- `cd agent && swift build`
- `cd agent && swift run StratoAgent`
- `cd agent && swift test`
- `cd agent && swift test --filter <TestCase>/<testName>`

### Shared Package (Swift)

- `cd shared && swift build`
- `cd shared && swift test`
- `cd shared && swift test --filter <TestCase>/<testName>`

### Libraries (Swift)

- `cd libs/swift-scim && swift build && swift test`
- `cd libs/swift-ssf && swift build && swift test`
- `cd libs/swift-qemu && swift build`
- `cd libs/swift-ovn && swift build`

### Docs (VitePress)

- `npm run docs:dev` — dev server for documentation
- `npm run docs:build` — build docs site
- `npm run docs:preview` — preview built docs

## Lint Commands

### Control Plane JS/CSS

- `cd control-plane && npm run lint`
- `cd control-plane && npm run lint:fix`
- `cd control-plane && npm test` (alias for lint)

## Test Notes

- SwiftPM supports `--filter` for single test runs:
  - Example: `swift test --filter UserControllerTests/testCreateUser`
  - Example: `swift test --filter UserControllerTests`
- Control-plane tests use in-memory SQLite when `app.environment == .testing`.
- Shared + agent packages do not require external services to test.

## Generated/Derived Files

- `control-plane/Public/styles/app.generated.css` is auto-generated by Tailwind.
- Avoid manual edits to generated outputs; change the source in
  `control-plane/Resources/styles/app.css` instead.

## Code Style Guidelines

### Swift Formatting

- Use 4-space indentation; no tabs.
- Keep braces on the same line as declarations.
- Prefer trailing commas in multiline argument lists and collections.
- Keep blank lines between logical blocks.
- Use `// MARK: -` sections for larger files and model groupings.

### Imports

- One import per line.
- Keep imports grouped together at the top of the file.
- Order imports consistently (roughly alphabetical is common here).

### Naming Conventions

- Types: `PascalCase` (e.g., `VMData`, `SchedulerService`).
- Methods/variables: `camelCase` (e.g., `configureWebAuthn`, `controlPlaneURL`).
- Acronyms stay uppercase in type names (`QEMUService`, `OIDCProvider`).
- Prefer descriptive names; avoid single-letter variables outside short scopes.

### Types and Concurrency

- Swift 6 strict concurrency is enabled.
- Use `Sendable` on shared models meant to cross concurrency boundaries.
- Prefer explicit types in public APIs and shared DTOs.

### Error Handling

- Use `guard` for early validation and clear exit paths.
- Log errors with `logger.error` and propagate failures via `throw`.
- Return explicit `ExitCode.failure` in CLI flows when appropriate.

### Swift API Design

- Favor `struct` for data models and DTOs.
- Use `public` access modifiers intentionally in shared packages.
- Keep initializers explicit when defaults matter.
- Prefer computed properties for derived values (see `VMData`).

### Control Plane (Vapor)

- Use middleware for cross-cutting concerns (auth, sessions, telemetry).
- Keep migrations registered in `configure.swift`.
- Environment variables should have sensible defaults for development.

### Agent

- Prefer `Logger` for operational output.
- Validate user input early and log intent clearly.
- Keep platform-specific implementations in separate files (`NetworkServiceLinux` vs `NetworkServiceMacOS`).

## File Structure Notes

- Control-plane server code: `control-plane/Sources/App/`
- Leaf templates: `control-plane/Resources/Views/`
- HTMX templates: `control-plane/web/templates/`
- Agent code: `agent/Sources/StratoAgent/`
- Shared models: `shared/Sources/StratoShared/`
- Libraries: `libs/swift-ovn/`, `libs/swift-qemu/`, `libs/swift-scim/`, `libs/swift-ssf/`

## Recommended Workflow

1. Choose the relevant package (`control-plane`, `agent`, `shared`).
2. Build and run tests locally via `swift build` / `swift test`.
3. Use `task dev` for full-stack integration testing.
4. Run lint for control-plane JS changes.

## Notes for Agentic Tools

- Do not run destructive git commands.
- Do not edit generated assets directly.
- Prefer minimal, focused changes consistent with local style.
